# AGENTS.md — KMS (Knowledge Management System)

Монорепозиторий прототипа веб-сервиса управления знаниями. Три изолированных Python-микросервиса, React-фронтенд, NGINX-шлюз.

## Архитектура и границы

| Сервис | Порт | База данных | Ответственность |
|--------|------|-------------|-----------------|
| `services/auth` | 8001 | `kms_auth` | Аутентификация, JWT, роли |
| `services/content` | 8002 | `kms_content` | Модули, уроки, эвристики (FAQ) |
| `services/assessment` | 8003 | `kms_assessment` | Тесты, вопросы (JSONB), попытки |
| `frontend/` | 5173 (dev) / 3000 (prod) | — | React + Vite + Axios |
| `infra/nginx/` | 80 (dev) | — | API Gateway, маршрутизация `/api/v1/*` → сервисы |

**Изоляция:** каждый сервис — только своя БД, только через NGINX. Прямые вызовы между сервисами запрещены.

**Роли:**
- `methodist` — создаёт контент, тесты, видит все результаты
- `seminarist` — проводит семинары, проходит тесты, пополняет FAQ
- `candidate` — изучает материалы, проходит тесты

## Управление зависимостями

- **Только `uv`**. `pip` напрямую запрещён.
- Каждый сервис — независимый проект со своим `pyproject.toml` и `uv.lock` (нет общего workspace).
- `uv.lock` коммитить вместе с `pyproject.toml`.

```bash
cd services/<service>
uv add <pkg>
uv add --dev pytest pytest-asyncio httpx
uv sync
uv run pytest
uv run uvicorn app.main:app --reload --port <PORT>
```

## База данных

- **Только async:** SQLAlchemy 2.x + `asyncpg`. Синхронные драйверы запрещены.
- Перед изменением схемы — миграция Alembic:
  ```bash
  uv run alembic revision --autogenerate -m "description"
  uv run alembic upgrade head
  ```

## Авторизация

- JWT (access + refresh). При выходе — JTI в Redis-blacklist с TTL = сроку жизни токена.
- Все эндпоинты защищать `Depends(get_current_user)` + проверкой роли.
- Валидацию blacklist выполнять через Redis в каждом микросервисе.

## Контент

- Файлы презентаций **не хранятся** на сервере. В БД — только URI-ссылки на «Р7-Офис», встраиваются через `<iframe>`.

## Секреты и окружение

- Жёстко прописывать секреты в коде запрещено. Только переменные окружения.
- Шаблон переменных — `.env.example` в корне.

## Git workflow

- **main** — только стабильный, проверенный код (merge из `develop` после 100% прохождения тестов).
- **develop** — основная ветка разработки, интеграция фич.
- **feature/*** — каждая фича/задача в отдельной ветке от `develop`.
  - Работа над каждым микросервисом, фронтендом или инфраструктурой — в **своей отдельной ветке** `feature/<название>`.
  - После завершения работы и прохождения всех тестов агент **обязан уведомить пользователя** и получить его утверждение перед слиянием.
  - Слияние в `develop` выполняется **только через Pull Request** после утверждения пользователем.
- **Коммит только рабочего состояния:** реализовать мелкую фичу → проверить код и тесты → коммит. Не коммитить поломанный или промежуточный код.
- **Не копить изменения.** Как только сделано одно логически завершённое действие — сразу коммит. Исправил опечатку — коммит, сделал фичу — коммит, обновил документацию — коммит. Мелкие частые коммиты предпочтительнее больших свалок изменений.
- Сообщения коммитов — на русском, отражают суть изменений.

## Стиль кода и линтинг

- **Линтер:** `ruff` — установлен в dev-зависимости каждого сервиса.
- **Конфигурация:** корневой `ruff.toml` с правилами для Python 3.13, async, security, docstrings.
- **Перед коммитом обязательно проверять:**
  ```bash
  cd services/<service>
  uv run ruff check app/ tests/
  uv run ruff format --check app/ tests/
  ```
- **Автоисправление:** `uv run ruff check app/ tests/ --fix`
- **Форматирование:** `uv run ruff format app/ tests/`
- **Требования:**
  - Длина строки ≤ 100 символов
  - Двойные кавычки
  - Google-style docstrings для всех публичных функций
  - Сортировка импортов (isort)
  - Проверки async (flake8-async)
  - Security checks (bandit)
  - Современный синтаксис Python 3.13 (pyupgrade)

## Тестирование

- **Каждая функция/эндпоинт покрывается автотестами.** Тесты пишутся до или сразу после реализации фичи, в той же ветке.
- Использовать `pytest` + `pytest-asyncio` + `httpx` для backend.
- Покрытие: happy path, граничные случаи, ошибки (404, 422, 401, 403).
- Тесты запускать через `uv run pytest`.
- **Цель:** покрыть автотестами бОльшую часть кода перед merge в `main`.
- **Делегирование:** написание автотестов для новых функций и эндпоинтов следует делегировать субагенту `test-writer` (см. `.opencode/agents/`).

## Порядок работы агента

1. Прочитать этот файл перед началом сессии.
2. Работать в рамках одного микросервиса — не переносить логику между сервисами.
3. Миграция перед изменением схемы → код → тесты → коммит.
4. Коммитить по одной логической единице с осмысленным сообщением. Не копить изменения — сделал фичу, проверил, сразу коммит.
5. Проверять написанный код и тесты перед коммитом (можно использовать браузер для тестирования).
6. При возникновении вопросов — задавать пользователю.

## REST конвенции

- **PUT** для полного обновления ресурса (включая `manager_id` для перевода пользователя).
- **PATCH** для частичного обновления (не рекомендуется для изменения manager_id).
- DELETE — только для админа или владельца ресурса.

## Иерархия пользователей и инвайты

- **Регистрация только по инвайту** (кроме первого пользователя — автоматически admin).
- Роли: `admin`, `methodist`, `seminarist`, `candidate`.
- `admin`: полный доступ, управляет всеми инвайтами.
- `methodist`: независимый (`manager_id=null`), может создавать инвайты для `candidate`/`seminarist`.
- `candidate`/`seminarist`: подчиняются `methodist` (поле `manager_id`).
- Перевод пользователя между менеджерами — через `PUT /users/{id}` с полем `manager_id`.

## Работа со скриншотами и артефактами

- Все скриншоты, сделанные агентом для анализа контекста (UI, ошибки, состояние страницы и т.д.),
  **обязательно сохранять в `logs/screenshots/`**.
- Создавать подпапки по контексту, например:
  - `logs/screenshots/auth/` — для скриншотов auth-сервиса
  - `logs/screenshots/frontend/` — для скриншотов фронтенда
  - `logs/screenshots/errors/` — для скриншотов ошибок
- Именовать файлы осмысленно: `YYYY-MM-DD_<краткое_описание>.png`.
- Не оставлять скриншоты в корне репозитория или в других случайных местах.
- **Удалять неактуальные артефакты немедленно.** Как только скриншот или любой другой артефакт потерял актуальность (даже частично) — сразу удалить файл. Не копить мусор в `logs/`.

## Аутентификация в микросервисах

- См. полную инструкцию по интеграции JWT и RBAC в новые микросервисы: [`docs/auth-integration.md`](docs/auth-integration.md).
